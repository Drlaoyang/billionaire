# 大富翁游戏（控制台版）架构设计文档

**版本**：1.0  
**目标平台**：命令行交互  
**图形界面**：暂不需要  
**文档性质**：纯架构与设计说明，不含代码实现

---

## 一、概述

本文档描述一款经典大富翁（Monopoly）游戏的逻辑架构设计方案。游戏基于控制台运行，支持2~6名玩家参与，实现移动、购地、建房、交易、卡牌事件、监狱、破产与胜负判定等核心机制。整体设计遵循模块化与分层解耦原则，便于后续扩展图形界面或AI玩家。

---

## 二、系统架构层次

系统自上而下划分为五个主要层次，各层职责明确、依赖方向清晰：

1. **入口层**  
   - 负责解析命令行参数、初始化游戏环境、启动游戏主循环。

2. **引擎层**  
   - 游戏主循环控制器，管理回合流转、玩家行动调度、胜负判定及游戏结束处理。

3. **逻辑核心层**  
   - 包含地图、玩家、卡牌堆三大核心模块。  
   - 地图模块管理40个格子的布局与索引；玩家模块存储个体状态与资产；卡牌堆模块管理机会与命运卡牌的抽取与生效。

4. **实体抽象层**  
   - 定义格子抽象基类及其派生类型，实现多态的土地触发事件。  
   - 包括可购买地产、车站、公用事业、税金格、监狱相关格、机会/命运格、免费停车格等。

5. **辅助服务层**  
   - 提供银行交易、拍卖管理、玩家间交易协调、骰子掷点等功能。

各模块之间通过游戏引擎进行协调，下层模块不直接依赖上层。

---

## 三、核心模块功能说明

### 3.1 游戏引擎

游戏引擎作为整个游戏的控制中枢，负责以下工作：

- **回合管理**  
  维护当前玩家索引，依次调用玩家行动逻辑。跳过已破产玩家。  
- **监狱特殊流程**  
  检查处于监狱状态的玩家，提供保释、使用出狱卡或掷双骰出狱的选择。  
- **移动与格子触发**  
  调用骰子模块获取点数，更新玩家位置，判断是否经过起点发放奖励，然后调用目标格子的落地事件。  
- **购买与拍卖调度**  
  当玩家停于无主地产时，询问购买意向；若拒绝则触发拍卖流程。  
- **破产处理与胜负判定**  
  玩家资产不足以支付债务时标记为破产，并移交资产给债权人。当仅剩一名未破产玩家时结束游戏。

### 3.2 地图模块

地图模块维护一个由40个格子对象组成的线性序列，提供按索引查找格子的能力。格子顺序遵循标准大富翁布局（起点、地产、车站、公用事业、机会、命运、税、监狱等）。

**主要职责**：

- 初始化标准棋盘布局（支持从配置文件读取）。  
- 提供给定索引返回对应格子实例的方法。  
- 管理特殊格子的固定索引（如监狱、起点位置），供引擎计算移动过圈奖励或送入监狱使用。

### 3.3 玩家模块

玩家模块封装单个玩家的完整状态信息，包括：

- **基础属性**：姓名、玩家编号、当前金钱、当前位置索引。  
- **资产列表**：持有的地产对象引用列表。  
- **监狱状态**：剩余监禁回合数、持有的出狱免费卡数量。  
- **状态标识**：正常、破产、在监狱中。

玩家模块还提供支付与收款的基础方法，用于处理金钱转移。当支付金额超过持有金钱时，需返回失败信号交由引擎触发破产流程。

### 3.4 格子抽象体系

所有格子均继承自一个抽象基类，该基类定义了统一的落地事件接口。不同类型的格子重写此接口实现特定行为：

- **可购买地产格**  
  - 属性：购买价格、所属色组、租金等级表（0~5级对应空地至旅馆）、房屋建造单价、当前房屋数、所有者、是否抵押。  
  - 落地逻辑：  
    - 若无主，提示购买。  
    - 若有主且未抵押，计算租金并要求支付。  
    - 租金计算需考虑同色系是否集齐、房屋数量等因素。

- **车站格**  
  - 租金随同玩家拥有的车站总数递增（1个25、2个50、3个100、4个200）。

- **公用事业格**  
  - 租金根据掷骰点数与拥有公用事业数量决定（1个为点数×4，2个为点数×10）。

- **机会与命运格**  
  - 落地时从对应卡牌堆抽取一张卡牌并立即执行其效果。

- **税金格**  
  - 落地时要求玩家支付固定金额或按比例计算的税款。税款可归银行或进入免费停车奖金池。

- **去监狱格**  
  - 立即将玩家位置设为监狱格索引，并标记为入狱状态。

- **免费停车格**  
  - 无实际效果，通常作为收集税金与罚款的奖金池发放点。

- **起点格**  
  - 经过或停留时可获得固定奖励，落地本身无额外事件。

- **监狱格**  
  - 停留在该格时无惩罚（仅当被送入监狱时状态变更）。

### 3.5 卡牌系统

卡牌系统由两个卡堆对象（机会卡堆、命运卡堆）及单张卡牌描述组成。

- **卡牌内容**  
  每张卡牌包含描述文本和对应的执行效果。执行效果定义为一组对玩家和游戏引擎的操作（如移动位置、增减金钱、出狱卡获取、房屋修缮费用等）。

- **卡牌堆管理**  
  - 支持初始化时洗牌。  
  - 抽卡时从牌堆顶部取走卡牌并放入弃牌堆。  
  - 当牌堆抽空时，将弃牌堆重新洗牌后转为新的牌堆，确保循环使用。

### 3.6 银行与交易辅助模块

**银行服务**  

- 管理地产的购买、抵押、赎回交易。  
- 在玩家放弃购买时主持拍卖流程。  
- 处理房屋的建造与出售（以半价回收）。  
- 负责税款、罚款等金钱的最终归集（部分可能流入免费停车池）。

**拍卖管理**  

- 当玩家拒绝购买无主地产时触发。  
- 所有未破产玩家按顺序叫价，起始价为地产面值的10%。  
- 每次加价至少为5元，直至无人加价则成交。

**玩家间交易**  

- 支持两名玩家自由协商交换现金、地产、出狱卡。  
- 系统仅验证双方是否拥有交易标的物，不限制交易公平性。

**骰子服务**  

- 模拟两颗六面骰子的掷点，返回点数及是否双骰。  
- 可用于检测连续双骰次数以执行入狱规则。

## 四、关键规则与流程设计

### 4.1 移动与过圈奖励

- 玩家每次掷两颗骰子，点数之和为移动步数。  
- 当新位置索引小于旧位置索引时（即绕过起点），自动获得200元过圈奖金。  
- 若掷出双骰，玩家可在本次行动结束后再掷一次；连续三次双骰则直接送入监狱。

### 4.2 监狱机制

- **入狱途径**：  
  - 停在“去监狱”格。  
  - 连续掷出三次双骰。  
  - 抽到直接入狱的卡牌。

- **出狱途径**：  
  - 支付50元保释金（可在任何回合开始时支付）。  
  - 使用一张“出狱免费卡”。  
  - 在监狱中尝试掷双骰，若成功则免费出狱；若连续三回合失败，第三回合结束时强制支付保释金出狱。

- **监狱中行为**：  
  - 可正常收取租金、参与拍卖与交易。  
  - 不能主动移动，但仍能通过卡牌效果移动。

### 4.3 租金计算规则

- **普通地产**：  
  - 若拥有整组同色地产且无房屋，租金翻倍。  
  - 租金根据房屋数查表（0~5级），旅馆为最高级。

- **车站**：  
  - 租金 = 25 × 2^(拥有车站数-1)（即1个25，2个50，3个100，4个200）。

- **公用事业**：  
  - 拥有1个：租金 = 掷骰点数 × 4。  
  - 拥有2个：租金 = 掷骰点数 × 10。

### 4.4 房屋建造与地产管理

- **建造条件**：  
  - 必须拥有某一色组全部地产。  
  - 该色组内各地产房屋数必须均衡（不能有任一地产比同组其他地产多出超过1栋房屋）。  
  - 不能超过旅馆上限（5栋）。

- **抵押与赎回**：  
  - 地产可抵押给银行获得面值一半的现金。抵押期间不能收取租金。  
  - 赎回时需支付抵押值外加10%的利息。  
  - 抵押地产不能建造房屋，必须先赎回。

### 4.5 破产与游戏结束

- 当玩家需支付金钱而现金不足时，进入破产处理流程：  
  - 向银行破产：所有地产归还银行，抵押状态解除。  
  - 向其他玩家破产：将所有资产（现金、地产、出狱卡）转移给债权人。  
- 当仅剩一名玩家未破产时，该玩家获胜，游戏结束。

## 五、扩展性考虑

### 5.1 配置外部化

地图数据、卡牌效果、地产价格与租金等信息建议存储于外部配置文件（如JSON或YAML）。通过读取配置初始化游戏，可在不修改代码的情况下调整游戏平衡或自定义地图。

### 5.2 命令模式与AI扩展

将玩家操作（掷骰、购买、出价等）设计为独立的命令对象，游戏引擎仅负责执行命令序列。这一设计便于：
- 实现游戏回放与撤销。  
- 为AI玩家接入决策算法，AI仅需生成对应的命令对象即可。

### 5.3 界面分离

所有输入输出均通过统一的IO接口进行，引擎不直接调用控制台函数。后续替换为图形界面或网络通信时，只需实现新的IO适配器，游戏核心逻辑零改动。

### 5.4 存档与读档

游戏状态可序列化为结构化数据（如JSON），包含所有玩家、地产、卡堆状态。引擎提供导出与导入接口即可实现存档功能。

## 六、总结

本文档从架构分层、模块职责、核心规则及扩展性四个方面描述了一款控制台大富翁游戏的设计方案。整体设计强调模块解耦与逻辑内聚，在满足当前控制台交互需求的同时，为后续图形化、网络化或AI化开发奠定了坚实基础。开发者可依据本设计快速进入实现阶段，并根据实际需要进一步细化各模块的内部数据结构与交互协议。
